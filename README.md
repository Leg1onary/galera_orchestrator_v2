**Galera Orchestrator v2** is a production-grade ops panel for teams running **MariaDB Galera** clusters.
No agents on nodes. No complex setup. Just a Docker container, your SSH key, and full control.

## рЯЪА Quick Start

### Prerequisites

| Dependency | Version |
|---|---|
| рЯР≥ Docker | 24+ |
| рЯР≥ Docker Compose | v2 plugin |
| рЯФС SSH private key | RSA / Ed25519, passwordless, access to all Galera nodes |

### 1. Clone & configure

```bash
git clone https://github.com/Leg1onary/galera_orchestrator_v2.git
cd galera_orchestrator_v2
cp .env.example .env
```

Open `.env` and **change all three secrets before first start**:

```env
ADMIN_USERNAME=admin
ADMIN_PASSWORD=your-strong-password        # вЖР change this

# Generate: openssl rand -hex 32
JWT_SECRET_KEY=<random-32+-chars>           # вЖР change this

# Generate: python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
FERNET_SECRET_KEY=<fernet-key>             # вЖР change this

SSH_KEY_HOST_PATH=/root/.ssh/id_rsa        # path to your SSH key on the host
```

> вЪ†пЄП `JWT_SECRET_KEY` and `FERNET_SECRET_KEY` must be **different** values.

### 2. Start

```bash
docker compose up -d
```

| | |
|---|---|
| рЯМН **Panel** | http://localhost:8000 |
| рЯС§ **Login** | `admin` / password from `.env` |
| рЯУЦ **API Docs** | http://localhost:8000/docs |

### 3. Add your first cluster

1. Go to **Settings вЖТ Clusters** вЖТ create cluster
2. **Settings вЖТ Datacenters** вЖТ create datacenter(s)
3. **Settings вЖТ Nodes** вЖТ add nodes (host, SSH port, DB credentials)
4. Select cluster in the top bar вЖТ watch **Overview** come alive рЯОЙ

### 4. Stop (data preserved)

```bash
docker compose down
```

## рЯФС SSH Key Setup

The orchestrator uses **one global SSH key** for all node/arbitrator connections.
The key is **bind-mounted read-only** into the container вАФ never stored in the database.

```yaml
# docker-compose.yml (already set up)
volumes:
  - ${SSH_KEY_HOST_PATH}:/root/.ssh/id_rsa:ro
```

вЬЕ Verify access before starting:

```bash
ssh -i ~/.ssh/id_rsa -p 22 root@<node-host> "hostname && mysql -e 'SELECT 1'"
```

Using a non-standard path? Just set it in `.env`:

```env
SSH_KEY_HOST_PATH=/home/ops/.ssh/galera_ed25519
```

> The key must be **passwordless**. `ssh-keygen -t ed25519 -N ""` creates one in seconds.

## рЯЦ•пЄП Pages & Features

### рЯП† Overview

The command center. Everything important at a glance.

- **Cluster summary bar** вАФ status, node count, primary component health
- **NodeCards** вАФ per-node state, wsrep metrics, read_only flag, maintenance badge
- **Sparklines** вАФ 30-point ring buffer for `flow_control_paused` and `recv_queue`
- **Event Log** вАФ real-time stream of cluster events with severity badges
- **WebSocket indicator** вАФ live connection status in footer

### рЯЧК Nodes

Full node table with everything you need for day-to-day ops.

- Sort/filter by name, state, datacenter, contour
- **NodeDetailDrawer** вАФ expand any node for full wsrep variable dump
- Per-node actions: toggle read_only, enter/exit maintenance, restart MariaDB
- **Connection test** вАФ on-demand SSH + DB reachability check
- State badges: `SYNCED` ¬Ј `JOINED` ¬Ј `DONOR` ¬Ј `DESYNCED` ¬Ј `OFFLINE`

### рЯЧЇпЄП Topology

SVG canvas вАФ visual representation of your cluster layout.

- Nodes and arbitrators grouped by **datacenter zones**
- **Contour** groupings (prod / staging / etc.)
- Connection lines with state color coding
- Zoom, pan, drag вАФ full interactive canvas
- Real-time state updates via WebSocket

### рЯЪС Recovery

Step-by-step wizard for full cluster recovery when **all nodes are down**.

```
Step 1 вАФ Scan nodes
  вФФвФА SSH into each node, read cluster status & wsrep state
  вФФвФА Detects which nodes are offline / non-primary

Step 2 вАФ Select bootstrap node
  вФФвФА Shows seqno from grastate.dat for each node
  вФФвФА Highlights the safe-to-bootstrap candidate
  вФФвФА Manual override with explicit confirmation

Step 3 вАФ Bootstrap & rejoin
  вФФвФА Executes bootstrap on selected node
  вФФвФА Monitors SYNCED state before rejoining next node
  вФФвФА Real-time progress log via WebSocket

Step 4 вАФ Done
  вФФвФА Cluster health confirmation
  вФФвФА Link back to Overview
```

> рЯЫ°пЄП **Cluster lock**: only one recovery operation at a time. Concurrent attempts вЖТ `409 Conflict`.

### рЯФІ Maintenance

Controlled maintenance operations without cluster downtime.

- **Enter / Exit Maintenance** вАФ sets `read_only = ON/OFF` per node via SSH + MariaDB
- **Rolling Restart Wizard** вАФ restarts MariaDB node-by-node:
  1. Choose restart order (drag to reorder)
  2. Each node: enter maintenance вЖТ `systemctl restart mariadb` вЖТ wait for `SYNCED` вЖТ next
  3. Zero downtime for the cluster as a whole
- Live progress log, abort at any step

> вЪ†пЄП Never put **all nodes** into maintenance simultaneously вАФ the cluster will lose quorum.

### рЯФН Diagnostics

SSH-powered cluster health inspection without leaving the browser.

- **Config diff** вАФ compare `my.cnf` / `galera.cnf` across all nodes, highlight differences
- **Variable check** вАФ verify critical Galera variables match across the cluster
- **Resource probe** вАФ disk usage, memory, open file handles via SSH
- Per-node execution вАФ run diagnostics on selected subset of nodes

### вЪЩпЄП Settings

Full CRUD management of all cluster entities.

| Entity | Fields |
|---|---|
| **Clusters** | Name, description, contour |
| **Datacenters** | Name, cluster association |
| **Nodes** | Host, ports, SSH user, DB credentials (Fernet-encrypted), datacenter |
| **Arbitrators** | Host, SSH port, datacenter |
| **System** | Polling interval, SSH/DB timeouts |

## рЯФД Real-time & WebSocket

All live data flows through a single WebSocket per cluster:

```
WS /ws/clusters/{cluster_id}
```

Authentication via the same `httpOnly` JWT cookie вАФ no separate WS token needed.

| Event | Payload | Triggered by |
|---|---|---|
| `node_state_changed` | Updated node live-fields | Poller detects state change |
| `arbitrator_state_changed` | Updated arbitrator state | Poller detects arb change |
| `operation_started` | Operation id, type, target | Recovery / Maintenance start |
| `operation_progress` | Step log line, timestamp | Mid-operation execution |
| `operation_finished` | Final status, duration | Operation completes or fails |
| `log_entry` | New event_log record | Any cluster event |

**Poller** runs as an asyncio background task, polling every `polling_interval_sec` seconds (default: 5s, configurable in Settings). Interval is read from DB on every cycle вАФ Settings changes take effect on next tick without restart.

## рЯТЊ Data Persistence & Backup

SQLite lives at `/data/orchestrator.db` inside the container, backed by the `orchestrator-data` named Docker volume. **Data survives container restarts and image upgrades.**

### Backup

```bash
docker run --rm \
  -v orchestrator-data:/data \
  -v "$(pwd)":/backup \
  alpine \
  tar czf /backup/orchestrator-db-$(date +%Y%m%d-%H%M%S).tar.gz /data
```

### Restore

```bash
docker compose down
docker run --rm \
  -v orchestrator-data:/data \
  -v "$(pwd)":/backup \
  alpine \
  sh -c "cd / && tar xzf /backup/orchestrator-db-YYYYMMDD-HHMMSS.tar.gz"
docker compose up -d
```

### Upgrade (data preserved)

```bash
git pull
docker compose build --no-cache
docker compose up -d   # volume untouched, container recreated
```

## рЯФТ Security Notes

| Concern | Mitigation |
|---|---|
| рЯФС **Admin password** | Change `ADMIN_PASSWORD` before first deploy вАФ default is `changeme` |
| рЯОЂ **JWT secret** | Generate with `openssl rand -hex 32`; `.env` only, never commit |
| рЯФР **Node DB passwords** | Encrypted at rest with Fernet вАФ changing `FERNET_SECRET_KEY` invalidates all stored passwords |
| рЯТї **SSH key** | Mounted `:ro` вАФ container cannot modify or exfiltrate it |
| рЯН™ **JWT in browser** | Lives only in `httpOnly` cookie вАФ inaccessible to JavaScript, XSS-safe |
| рЯФЗ **Dual secrets** | `JWT_SECRET_KEY` and `FERNET_SECRET_KEY` **must** be different values |
| вЪ†пЄП **No mock mode** | All SSH / SQL operations are **real** вАФ use a test cluster for experiments |
| рЯМР **CORS** | In production, SPA and API are on the same origin вАФ CORS middleware active in dev only |

## рЯФІ Troubleshooting

<details>
<summary>рЯЪЂ Panel unreachable after <code>docker compose up</code></summary>

```bash
docker compose logs orchestrator    # check startup errors
docker compose ps                   # verify container is running
```
</details>

<details>
<summary>рЯФР <code>401 Unauthorized</code> after login</summary>

Make sure you access the panel on the **same host and port** configured in `HOST_PORT`.
`httpOnly` cookies are not sent cross-origin вАФ accessing via IP when configured for hostname (or vice versa) will break auth.
</details>

<details>
<summary>рЯФі Node immediately shows <code>OFFLINE</code></summary>

```bash
# Test SSH from the host
ssh -i ~/.ssh/id_rsa -p <ssh_port> <ssh_user>@<node_host> "hostname"

# Test DB
mysql -h <node_host> -P <db_port> -u <db_user> -p -e "SELECT 1"
```
</details>

<details>
<summary>рЯФР Node DB passwords broken after config change</summary>

If you changed `FERNET_SECRET_KEY`, all encrypted passwords are now unreadable.
Go to **Settings вЖТ Nodes** and re-enter `db_password` for every affected node.
</details>

<details>
<summary>рЯУі WebSocket shows <code>Disconnected</code> in footer</summary>

```bash
docker compose ps    # confirm container is up

# Manual WS upgrade test
curl -i -N \
  -H "Upgrade: websocket" \
  -H "Connection: Upgrade" \
  -H "Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==" \
  -H "Sec-WebSocket-Version: 13" \
  http://localhost:8000/ws/clusters/1
```
</details>

<details>
<summary>рЯФД Rolling Restart stuck mid-way</summary>

If a rolling restart is interrupted, the affected node stays in maintenance (`read_only = ON`).
Go to **Maintenance вЖТ Node Maintenance State** вЖТ find the node вЖТ click **Exit**.
This releases `read_only` without requiring a full restart.
</details>

<details>
<summary>вЪЩпЄП Settings changes not reflected in polling</summary>

Polling interval and SSH/DB timeouts are read from the database on every poll cycle.
No restart needed вАФ changes take effect on the next tick (within `polling_interval_sec` seconds).
</details>
