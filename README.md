## вҡЎ What is this?

**Galera Orchestrator v2** is a production-grade ops panel for teams running **MariaDB Galera** clusters.
No agents on nodes. No complex setup. Just a Docker container, your SSH key, and full control.

## рҹ”‘ SSH Key Setup

The orchestrator uses **one global SSH key** for all node/arbitrator connections.
The key is **bind-mounted read-only** into the container вҖ” never stored in the database.

```yaml
# docker-compose.ghcr.yml (already configured)
volumes:
  - ${SSH_KEY_PATH}:/root/.ssh/id_rsa:ro
```

вң… РҹСҖРҫРІРөСҖСҢ РҙРҫСҒСӮСғРҝ РҝРөСҖРөРҙ Р·Р°РҝСғСҒРәРҫРј:

```bash
ssh -i ~/.ssh/id_rsa -p 22 root@<node-host> "hostname && mysql -e 'SELECT 1'"
```

Р•СҒР»Рё РәР»СҺСҮР° РҪРөСӮ вҖ” СҒРҫР·РҙР°Р№:

```bash
ssh-keygen -t ed25519 -N "" -f ~/.ssh/galera_key
ssh-copy-id -i ~/.ssh/galera_key.pub user@node-host
```

> РҡР»СҺСҮ РҙРҫР»Р¶РөРҪ РұСӢСӮСҢ **РұРөР· passphrase**.

- **WebSocket indicator** вҖ” live connection status in footer

### рҹ—Ҡ Nodes

Full node table with everything you need for day-to-day ops.

- Sort/filter by name, state, datacenter, contour
- **NodeDetailDrawer** вҖ” expand any node for full wsrep variable dump
- Per-node actions: toggle read_only, enter/exit maintenance, restart MariaDB
- **Connection test** вҖ” on-demand SSH + DB reachability check
- State badges: `SYNCED` В· `JOINED` В· `DONOR` В· `DESYNCED` В· `OFFLINE`

### рҹ—әпёҸ Topology

SVG canvas вҖ” visual representation of your cluster layout.

- Nodes and arbitrators grouped by **datacenter zones**
- **Contour** groupings (prod / staging / etc.)
- Connection lines with state color coding
- Zoom, pan, drag вҖ” full interactive canvas
- Real-time state updates via WebSocket

### рҹҡ‘ Recovery

Step-by-step wizard for full cluster recovery when **all nodes are down**.

```
Step 1 вҖ” Scan nodes
  в””в”Җ SSH into each node, read cluster status & wsrep state
  в””в”Җ Detects which nodes are offline / non-primary

Step 2 вҖ” Select bootstrap node
  в””в”Җ Shows seqno from grastate.dat for each node
  в””в”Җ Highlights the safe-to-bootstrap candidate
  в””в”Җ Manual override with explicit confirmation

Step 3 вҖ” Bootstrap & rejoin
  в””в”Җ Executes bootstrap on selected node
  в””в”Җ Monitors SYNCED state before rejoining next node
  в””в”Җ Real-time progress log via WebSocket

Step 4 вҖ” Done
  в””в”Җ Cluster health confirmation
  в””в”Җ Link back to Overview
```

> рҹӣЎпёҸ **Cluster lock**: only one recovery operation at a time. Concurrent attempts вҶ’ `409 Conflict`.

### рҹ”§ Maintenance

Controlled maintenance operations without cluster downtime.

- **Enter / Exit Maintenance** вҖ” sets `read_only = ON/OFF` per node via SSH + MariaDB
- **Rolling Restart Wizard** вҖ” restarts MariaDB node-by-node:
  1. Choose restart order (drag to reorder)
  2. Each node: enter maintenance вҶ’ `systemctl restart mariadb` вҶ’ wait for `SYNCED` вҶ’ next
  3. Zero downtime for the cluster as a whole
- Live progress log, abort at any step

> вҡ пёҸ Never put **all nodes** into maintenance simultaneously вҖ” the cluster will lose quorum.

### рҹ”Қ Diagnostics

SSH-powered cluster health inspection without leaving the browser.

- **Config diff** вҖ” compare `my.cnf` / `galera.cnf` across all nodes, highlight differences
- **Variable check** вҖ” verify critical Galera variables match across the cluster
- **Resource probe** вҖ” disk usage, memory, open file handles via SSH
- Per-node execution вҖ” run diagnostics on selected subset of nodes

| Entity | Fields |
|---|---|
| **Clusters** | Name, description, contour |
| **Datacenters** | Name, cluster association |
| **Nodes** | Host, ports, SSH user, DB credentials (Fernet-encrypted), datacenter |
| **Arbitrators** | Host, SSH port, datacenter |
| **System** | Polling interval, SSH/DB timeouts |

## рҹ”„ Real-time & WebSocket

All live data flows through a single WebSocket per cluster:

```
WS /ws/clusters/{cluster_id}
```

Authentication via the same `httpOnly` JWT cookie вҖ” no separate WS token needed.

| Event | Payload | Triggered by |
|---|---|---|
| `node_state_changed` | Updated node live-fields | Poller detects state change |
| `arbitrator_state_changed` | Updated arbitrator state | Poller detects arb change |
| `operation_started` | Operation id, type, target | Recovery / Maintenance start |
| `operation_progress` | Step log line, timestamp | Mid-operation execution |
| `operation_finished` | Final status, duration | Operation completes or fails |
| `log_entry` | New event_log record | Any cluster event |

**Poller** runs as an asyncio background task, polling every `polling_interval_sec` seconds (default: 5s, configurable in Settings). Interval is read from DB on every cycle вҖ” Settings changes take effect on next tick without restart.

## рҹ’ҫ Data Persistence & Backup

SQLite Р¶РёРІС‘СӮ РІ `/data/orchestrator.db` РІРҪСғСӮСҖРё РәРҫРҪСӮРөР№РҪРөСҖР°, Р·Р° РәРҫСӮРҫСҖСӢРј СҒСӮРҫРёСӮ named Docker volume `orchestrator-data`. **Р”Р°РҪРҪСӢРө РҝРөСҖРөР¶РёРІР°СҺСӮ РҝРөСҖРөР·Р°РҝСғСҒРә РәРҫРҪСӮРөР№РҪРөСҖР° Рё РҫРұРҪРҫРІР»РөРҪРёСҸ РҫРұСҖР°Р·Р°.**

### Backup

```bash
docker run --rm \
  -v orchestrator-data:/data \
  -v "$(pwd)":/backup \
  alpine \
  tar czf /backup/orchestrator-db-$(date +%Y%m%d-%H%M%S).tar.gz /data
```

### Restore

```bash
docker compose down
docker run --rm \
  -v orchestrator-data:/data \
  -v "$(pwd)":/backup \
  alpine \
  sh -c "cd / && tar xzf /backup/orchestrator-db-YYYYMMDD-HHMMSS.tar.gz"
docker compose up -d
```

### Upgrade

```bash
# Р РөРәРҫРјРөРҪРҙСғРөРјСӢР№ СҒРҝРҫСҒРҫРұ вҖ” СҮРөСҖРөР· update.sh
```

## рҹ”’ Security Notes

| Concern | Mitigation |
|---|---|
| рҹ”‘ **Admin password** | Change `ADMIN_PASSWORD` before first deploy вҖ” default is `changeme` |
| рҹҺ« **JWT secret** | Generate with `openssl rand -hex 32`; `.env` only, never commit |
| рҹ”җ **Node DB passwords** | Encrypted at rest with Fernet вҖ” changing `FERNET_SECRET_KEY` invalidates all stored passwords |
| рҹ’» **SSH key** | Mounted `:ro` вҖ” container cannot modify or exfiltrate it |
| рҹҚӘ **JWT in browser** | Lives only in `httpOnly` cookie вҖ” inaccessible to JavaScript, XSS-safe |
| рҹ”Ү **Dual secrets** | `JWT_SECRET_KEY` Рё `FERNET_SECRET_KEY` **must** be different values |
| вҡ пёҸ **No mock mode** | All SSH / SQL operations are **real** вҖ” use a test cluster for experiments |
| рҹҢҗ **CORS** | In production, SPA and API are on the same origin вҖ” CORS middleware active in dev only |

## рҹ”§ Troubleshooting

<details>
<summary>рҹҡ« Panel unreachable after <code>docker compose up</code></summary>

```bash
docker compose logs orchestrator    # check startup errors
docker compose ps                   # verify container is running
```
</details>

<details>
<summary>рҹ”җ <code>401 Unauthorized</code> after login</summary>

Make sure you access the panel on the **same host and port** configured in `HOST_PORT`.
`httpOnly` cookies are not sent cross-origin вҖ” accessing via IP when configured for hostname (or vice versa) will break auth.
</details>

<details>
<summary>рҹ”ҙ Node immediately shows <code>OFFLINE</code></summary>

```bash
# Test SSH from the host
ssh -i ~/.ssh/id_rsa -p <ssh_port> <ssh_user>@<node_host> "hostname"

# Test DB
mysql -h <node_host> -P <db_port> -u <db_user> -p -e "SELECT 1"
```
</details>

<details>
<summary>рҹ”җ Node DB passwords broken after config change</summary>

If you changed `FERNET_SECRET_KEY`, all encrypted passwords are now unreadable.
Go to **Settings вҶ’ Nodes** and re-enter `db_password` for every affected node.
</details>

<details>
<summary>рҹ“ҙ WebSocket shows <code>Disconnected</code> in footer</summary>

```bash
docker compose ps    # confirm container is up

# Manual WS upgrade test
curl -i -N \
  -H "Upgrade: websocket" \
  -H "Connection: Upgrade" \
  -H "Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==" \
  -H "Sec-WebSocket-Version: 13" \
  http://localhost:8000/ws/clusters/1
```
</details>

<details>
<summary>рҹ”„ Rolling Restart stuck mid-way</summary>

If a rolling restart is interrupted, the affected node stays in maintenance (`read_only = ON`).
Go to **Maintenance вҶ’ Node Maintenance State** вҶ’ find the node вҶ’ click **Exit**.
This releases `read_only` without requiring a full restart.
</details>

<details>
<summary>вҡҷпёҸ Settings changes not reflected in polling</summary>

Polling interval and SSH/DB timeouts are read from the database on every poll cycle.
No restart needed вҖ” changes take effect on the next tick (within `polling_interval_sec` seconds).
</details>
